# AI咨询知识速查

## 一、AI赋能网安（AI for Security）

### 1. AI驱动威胁检测
- **UEBA（用户实体行为分析）**：基于ML建立用户/实体行为基线，检测偏离。关键指标：登录时间异常、数据访问量突增、特权操作频率。常用算法：Isolation Forest、Autoencoder
- **恶意软件检测**：静态分析（PE头特征+NLP分析字符串）+ 动态分析（沙箱行为序列ML分类）。注意：对抗样本可使模型误判，需要对抗训练
- **DNS隧道检测**：基于DNS查询频率、域名熵值、查询类型分布的异常检测
- **加密流量分析**：TLS握手特征（JA3/JA4指纹）+ 流量包大小/时间序列模式 → 不解密识别恶意流量

### 2. AI驱动安全运营（AI for SOC）
- **LLM辅助日志分析**：自然语言查询日志（如「过去24小时有哪些来自境外IP的SSH爆破失败」→ 自动转Splunk/KQL查询）。关键：Prompt模板设计、结果验证机制
- **告警降噪**：ML对告警聚类+关联分析（同一攻击链的告警合并为1条事件）。典型降噪率70-90%
- **SOAR智能化**：LLM辅助生成Playbook（自然语言描述场景→自动生成SOAR剧本代码）。人工审核后再上线
- **威胁狩猎**：LLM分析MITRE ATT&CK TTP → 根据环境特征建议狩猎假设 → 自动生成狩猎查询
- **Copilot模式**：NLP转SIEM/KQL查询 + 自动生成事件分析摘要 + 推荐下一步调查方向

### 3. AI驱动代码安全（AI for Code Security）
- **LLM辅助SAST**：用LLM理解业务逻辑上下文 → 减少传统SAST的误报（如判断一个SQL拼接是否真的存在用户输入路径）
- **漏洞自动修复**：LLM对SAST发现的漏洞生成修复建议+代码补丁。需人工Review后合入。关键：Prompt中提供完整上下文（调用链+数据流）
- **智能合约审计**：针对Solidity/Rust合约的常见漏洞（重入、溢出、闪电贷）进行AI辅助检测
- **Prompt设计要点**：提供漏洞类型的精确定义+正反面代码示例+修复约束（不改变业务逻辑）

### 4. AI驱动渗透测试（AI for Pentest）
- **自动化信息收集**：LLM分析目标网站内容 → 自动生成攻击面清单（API端点、技术栈、潜在敏感路径）
- **Payload智能生成**：根据WAF拦截特征自动变异Payload，绕过规则。适用于SQL注入/XSS/命令注入
- **API Fuzzing**：LLM理解API文档/OpenAPI Spec → 自动生成边界值、异常类型、业务逻辑异常输入
- **漏洞报告生成**：自动将工具输出+手工测试记录转为结构化报告（参考templates/pentest-report.md）
- **限制说明**：AI渗透测试工具仍处于辅助阶段，不能替代人工判断。LLM对0day漏洞利用路径的创造力不如经验丰富的渗透工程师

### 5. AI驱动威胁情报（AI for Threat Intelligence）
- **NLP IOC提取**：自动从安全博客/漏洞公告/暗网帖子中提取IP/域名/URL/哈希/MITRE T编号
- **攻击者归因**：NLP分析勒索信/攻击声明/工具命名习惯 → 关联已知APT组织特征
- **多语言情报**：LLM翻译+分析非英语威胁情报来源（俄语/中文/波斯语/朝鲜语）
- **自动化情报简报**：每日/每周自动汇总相关威胁 + 根据客户行业过滤 → 3段话简报

### 6. AI驱动安全培训（AI for Training）
- **个性化钓鱼演练**：LLM根据员工岗位/工作内容/社交信息生成个性化钓鱼邮件 → 提高演练真实度。需脱敏
- **AI辅助CTF出题**：描述考点 → LLM生成题目+源码+Dockerfile+Writeup
- **自适应安全培训**：根据员工钓鱼演练结果/安全事件参与情况 → 推荐个性化培训内容
- **安全知识问答机器人**：基于企业安全制度/行业标准训练 → 员工可以自然语言提问安全政策

## 二、安全赋能AI（Security for AI）

### 1. AI系统威胁模型

#### OWASP Top 10 for LLM Applications (2025)
1. **LLM01: Prompt Injection（提示注入）** — 直接注入/间接注入（通过外部数据源）
2. **LLM02: Sensitive Information Disclosure（敏感信息泄露）** — 训练数据记忆/推理阶段信息泄露
3. **LLM03: Supply Chain（供应链）** — 第三方模型/插件/数据集/依赖风险
4. **LLM04: Data and Model Poisoning（数据与模型投毒）** — 训练/微调数据投毒、模型篡改
5. **LLM05: Improper Output Handling（不当输出处理）** — LLM输出未校验直接执行/渲染
6. **LLM06: Excessive Agency（过度代理权）** — Agent权限过大/操作未经人工确认
7. **LLM07: System Prompt Leakage（系统提示词泄露）** — 系统提示词中的密钥/控制逻辑被提取
8. **LLM08: Vector and Embedding Weaknesses（向量与嵌入弱点）** — RAG知识库投毒/跨租户数据泄露
9. **LLM09: Misinformation（错误信息）** — 幻觉输出被当作事实、人员过度依赖LLM
10. **LLM10: Unbounded Consumption（无限制消耗）** — 资源耗尽/模型拒绝服务/通过API查询蒸馏窃取模型

> 版本说明：本清单为OWASP 2025版。与2023 v1.1版的主要差异：LLM02由Insecure Output Handling改为敏感信息泄露（输出处理调整为LLM05）；原LLM07 Insecure Plugin Design并入供应链与过度代理权；新增LLM07系统提示词泄露、LLM08向量与嵌入弱点；原LLM10 Model Theft的蒸馏场景并入LLM10。引用时注意版本，避免新旧条目混用。

#### MITRE ATLAS（对抗性AI威胁矩阵）
- 与ATT&CK映射：侦察→资源开发→初始访问→ML模型访问→执行→持久化→防御规避→发现→收集→ML数据渗出→影响

### 2. 对抗攻击与防御

#### 攻击类型
- **提示注入（Prompt Injection）**：直接注入（"Ignore previous instructions…"）、间接注入（网页/文档内容中嵌入指令）
- **越狱（Jailbreaking）**：DAN角色扮演、Token Smuggling、多语言绕过、编码绕过（Base64/ROT13）
- **对抗样本（Adversarial Examples）**：像素级扰动、补丁攻击、物理世界对抗
- **模型窃取（Model Extraction）**：API查询蒸馏、侧信道攻击、成员推理攻击
- **数据投毒（Data Poisoning）**：后门攻击、标签翻转、可用性投毒
- **供应链攻击**：恶意模型（HuggingFace上的恶意Pickle文件）、被劫持的模型依赖

#### 防御措施
- **输入安全**：输入过滤（正则+LLM分类器双保险）、输入截断（限制长度）、语义审查
- **输出安全**：输出格式校验、内容安全API二次审查、敏感信息正则扫描
- **模型层面**：RLHF安全对齐、红队测试（自动化+人工）、对抗训练、模型水印
- **架构层面**：AI网关（统一鉴权+限流+内容过滤）、Human-in-the-loop关键操作
- **监控层面**：Prompt/Response日志（保留用于事后审计）、异常查询模式告警、越狱尝试检测

### 3. AI合规全景

#### 中国法规
- **《生成式人工智能服务管理暂行办法》**（2023.8.15施行）
  - 适用范围：向境内公众提供生成式AI服务的组织和个人
  - 核心义务：训练数据合法来源、内容安全、算法备案、用户实名（涉深度合成时）
  - 安全评估：具有舆论属性或社会动员能力的生成式AI服务需按国家有关规定开展安全评估
  - 算法备案：依据《互联网信息服务算法推荐管理规定》第24条，具有舆论属性或社会动员能力的服务提供者应在**提供服务之日起10个工作日内**，通过互联网信息服务算法备案系统（beian.cac.gov.cn）办理备案；省级网信部门初审、国家网信办复审，材料齐全的30个工作日内予以备案并公示（第25条）
- **《互联网信息服务深度合成管理规定》**（2023.1.10施行）
  - 深度合成内容需显著标识（不影响用户使用的前提下）
  - 建立辟谣机制
- **TC260-003《生成式人工智能服务安全基本要求》**
  - 训练数据安全：数据来源可追溯、数据清洗、数据去重、个人信息保护
  - 模型安全：内容安全（涉政/涉黄/涉暴检测准确率>95%）、鲁棒性、可解释性
  - 服务安全：用户身份验证、不良信息过滤、安全事件应急响应
- **个人信息保护法**：使用个人信息训练模型需满足合法性基础（同意/去标识化/匿名化）

#### 国际法规
- **EU AI Act**（2024）：风险分级（不可接受/高/有限/最小风险），高风险AI系统需CE标志
- **NIST AI RMF v1.0**：Govern-Map-Measure-Manage四阶段框架
- **ISO/IEC 42001:2023**：AI管理体系标准（AIMS）

### 4. 关键术语速查
- RAG（检索增强生成）、Agent（智能体）、MCP（模型上下文协议）、Function Calling（函数调用）
- RLHF（基于人类反馈的强化学习）、SFT（监督微调）、DPO（直接偏好优化）

- Fine-tuning（微调）、Prompt Engineering（提示词工程）、Few-shot/Zero-shot
- Token（令牌/词元）、Context Window（上下文窗口）、Temperature（温度参数）
- MOE（混合专家架构）、Transformer、Attention（注意力机制）
- JA3/JA4（TLS指纹，用于加密流量分析和恶意软件检测）

## 三、AI方案落地检查清单

在给客户提供AI相关方案前，逐一确认：

### AI赋能网安方案检查
- [ ] 是否明确了AI解决的具体安全问题？（不写「提升安全水平」这种话）
- [ ] 是否有性能指标？（检测率/误报率/告警降噪率/响应时间）
- [ ] 是否说明了数据需求？（需要什么日志/多少历史数据/标注需求）
- [ ] 是否说明了与现有安全工具的集成方式？（SIEM/SOAR/EDR的接口）
- [ ] 是否说明了人的角色？（AI辅助决策还是AI自动处置？人工在哪个环节介入？）
- [ ] 是否评估了AI自身的风险？（对抗样本/模型漂移/告警疲劳）

### AI安全评估方案检查
- [ ] 是否覆盖了OWASP Top 10 for LLM的检查项？
- [ ] 是否包含了红队测试（自动化+人工）？
- [ ] 是否评估了训练数据/微调数据的合规性？
- [ ] 是否评估了AI供应链风险（第三方模型/插件）？
- [ ] 是否包含了AI合规（算法备案/安全评估/内容标识）？

### AI战略/选型方案检查
- [ ] 是否做了Make vs Buy分析（自研 vs 采购 vs API调用）？
- [ ] 是否对比了至少2个方案的总拥有成本（TCO）？
- [ ] 是否考虑了数据主权和数据跨境？（尤其是使用海外API的场景）
- [ ] 是否规划了AI人才/团队建设路径？
- [ ] 是否说明了「不建议用AI」的场景？（不是所有问题都需要AI解决）

# CSCA 质量审查标准

## 审查机制总览

CSCA 审查分**两轮**，必须按顺序执行：

```
第一轮：关键点事实核查 → 逐个关键断言验证 → 不通过则修正 → 通过后进第二轮
第二轮：整体质量审查 → 7维度评分 → 不达标则重写 → 通过后交付
```

---

# 第一轮：关键点事实核查（Key-Point Verification）

## 目的
确保输出的核心论点、数据、标准引用不存在硬伤。这一轮关注「对不对」，不关注「好不好」。

## 执行步骤

### 步骤1：提取关键断言
从交付物中提取所有「如果一个读者只记住5件事，应该记住什么」——就是这5-10个关键断言。

关键断言的特征：
- 引用了具体的标准条款（如「GB/T 22239-2019 第8.1.3条要求…」）
- 给出了具体的数据（如「检测到234个活跃IP」「修复成本约85万」）
- 做出了明确的结论（如「方案A优于方案B」「该漏洞可导致数据库完全泄露」）
- 定义了范围边界（如「仅覆盖互联网业务系统」）
- 涉及了法规合规判断（如「该系统需满足等保三级」）

### 步骤2：逐条验证
对每个关键断言进行以下检查：

| 验证维度 | 检查问题 | 不通过处理 |
|---------|---------|-----------|
| 标准引用 | 引用的标准名称、版本号、条款号是否正确？ | 查对原标准，修正 |
| 数据一致性 | 同一数据在不同位置是否一致？（如前面写234个IP，后面写约230个） | 统一数据 |
| 计算正确性 | 涉及的计算（合计/百分比/人天汇总）是否正确？ | 重新计算，修正 |
| 逻辑完备性 | 结论的前提条件是否成立？推导链条有没有跳跃？ | 补充推导步骤或标注假设 |
| 合规判断 | 法规适用判断是否正确？（如判断系统需要等保三级——定级标准对吗？） | 对照法规原文，修正 |
| 技术事实 | 技术描述是否有事实性错误？（如「Redis默认端口6379」是错的吗？「TLS 1.3使用AES-256-GCM」对吗？） | 核实后修正 |

### 步骤3：标记验证结果
每项验证输出以下三种结论之一：
- ✅ **已核实**：断言正确，可进入第二轮
- ⚠️ **已标注假设**：因为信息不足无法核实，已在文中明确标注为「基于以下假设…」
- ❌ **发现错误**：断言有误，已修正为…

### 第一轮输出格式
```
## 第一轮审查：关键点事实核查

| # | 关键断言 | 验证维度 | 结果 | 说明 |
|---|---------|---------|------|------|
| 1 | [断言摘要] | 标准引用 | ✅ | GB/T 22239-2019 8.1.3条确认正确 |
| 2 | [断言摘要] | 数据一致性 | ⚠️ | 已标注为基于客户提供数据的估算 |
| 3 | [断言摘要] | 计算正确性 | ❌→✅ | 合计金额修正为420万（原为415万） |

核查结论：[全部通过 / N项修正后通过]
```

# 第二轮：整体质量审查（7-Dimension Quality Review）

## 七维度评分体系

### 维度1：专业性（权重20%）

**评分标准**：
| 分数 | 标准 |
|------|------|
| 10 | 术语精准，方法论引用正确，行业规范应用恰当，无概念性错误 |
| 8-9 | 核心概念准确，偶有不影响理解的小偏差 |
| 7 | 主要概念正确，个别术语使用不够精确 |
| 5-6 | 存在1-2处概念混淆或术语错误 |
| 3-4 | 多处专业错误，影响方案可信度 |
| 1-2 | 大量概念性错误，方案不可用 |

**检查要点**：
- 安全术语使用是否正确（威胁 vs 风险 vs 漏洞的区分）
- 标准/法规引用是否准确（版本号、条款号是否对应）
- 技术方案是否与行业公认实践一致
- 安全控制措施是否逻辑自洽

---

### 维度2：逻辑性（权重15%）

**评分标准**：
| 分数 | 标准 |
|------|------|
| 10 | 问题→原因→方案→验证的推理链完整且可追溯 |
| 8-9 | 逻辑清晰，个别推理跳跃但不影响整体 |
| 7 | 主要逻辑链完整，次要论证略弱 |
| 5-6 | 存在1-2处逻辑断裂或循环论证 |
| 3-4 | 多处逻辑不一致，方案难以自圆其说 |
| 1-2 | 基本逻辑混乱，结论无支撑 |

**检查要点**：
- 问题→原因→方案→验证的逻辑链是否完整
- 是否存在循环论证或倒果为因
- 结论是否有足够的论据支撑
- 方案选择是否有对比分析（至少一个替代方案）
- 不同章节之间是否有逻辑矛盾

---

### 维度3：行业适配（权重15%）

**评分标准**：
| 分数 | 标准 |
|------|------|
| 10 | 深度结合客户行业特征，体现对该行业的深刻理解 |
| 8-9 | 行业特征识别准确，方案有明显行业定制痕迹 |
| 7 | 考虑了行业基本特征，方案不是通用模板 |
| 5-6 | 行业识别不够精准，部分建议不适合该行业 |
| 3-4 | 方案基本是通用内容，缺乏行业定制 |
| 1-2 | 完全无视行业特征，直接套用模板 |

**检查要点**：
- 是否针对客户行业进行了特定定制
- 是否考虑了该行业的特殊监管/合规要求
- 是否使用了该行业熟悉的术语体系
- 方案是否考虑了该行业的典型预算/时间约束

---

### 维度4：可实施性（权重20%）

**评分标准**：
| 分数 | 标准 |
|------|------|
| 10 | 实施方案清晰完整，资源明确，风险可控，直接可执行 |
| 8-9 | 实施路径清晰，资源估算基本合理 |
| 7 | 有实施步骤和资源需求，可执行 |
| 5-6 | 实施路径模糊，缺少关键资源估算 |
| 3-4 | 方案停留在方向层面，无法直接实施 |
| 1-2 | 纯理论，无实施可能性 |

**检查要点**：
- 是否提供了分阶段的实施路径（时间线、里程碑）
- 是否明确了资源需求（人员技能、工具、预算范围）
- 是否定义了验收标准（怎样算做完、怎样算做好）

---

### 维度5：完整性（权重10%）

**评分标准**：
| 分数 | 标准 |
|------|------|
| 10 | 覆盖需求全部范围，有引言有总结，附录齐全 |
| 8-9 | 主要内容完整，次要细节有少量遗漏 |
| 7 | 核心内容完整，边缘场景未充分覆盖 |
| 5-6 | 缺失1-2个重要方面的内容 |
| 3-4 | 多个关键内容缺失 |
| 1-2 | 大部分内容缺失，交付物无价值 |

**检查要点**：
- 是否覆盖了需求定义的全部范围
- 是否有引言和总结
- 是否有版本信息和适用范围标注

---

### 维度6：可维护性（权重5%）

**评分标准**：
| 分数 | 标准 |
|------|------|
| 10 | 结构清晰易于修订，有更新机制说明 |
| 8-9 | 结构合理，便于后续修改 |
| 7 | 基本可维护，结构无明显问题 |
| 5-6 | 结构混乱或过度复杂，修改成本高 |
| 3-4 | 难以维护，修改一处可能影响多处 |
| 1-2 | 无结构可言，不可维护 |

---

### 维度7：去AI程度（权重15%）

**评分标准**：
| 分数 | 标准 |
|------|------|
| 10 | 读起来完全是资深顾问手笔，无任何AI痕迹 |
| 8-9 | 自然专业，偶有一两处略显模板化 |
| 7 | 基本自然，无明显AI痕迹，细微处可优化 |
| 5-6 | 存在明显AI特征（禁用词、模板化结构、单一节奏） |
| 3-4 | AI痕迹明显，多处禁用词汇和机械表达 |
| 1-2 | 一眼AI，大量空话套话，无人类写作特征 |

**检查要点**：
- 是否存在writing.md B1中的120+禁用词汇
- 段落长度和句子长度是否自然多样
- 表格是否符合B3表格去AI化规则
- 反向测试：读完前两页能否判断出是AI写的？如果能 → 此项不合格

---

## 第二轮执行协议

### 步骤1：逐维度评分
对每个维度打分（1-10），记录扣分项和具体问题。

### 步骤2：计算加权综合得分
按维度权重（专业性20%、逻辑性15%、行业适配15%、可实施性20%、完整性10%、可维护性5%、去AI程度15%）计算加权综合得分（满分10分），作为交付质量的整体评价记录在审查块中。

### 步骤3：判断通过/不通过
- 通过门槛：所有维度 ≥ 7。这是单项门槛，不用加权分替代——任一短板维度都必须重写
- 所有维度 ≥ 7 → 通过，输出审查块并交付
- 任一维度 < 7 → 进入步骤4

### 步骤4：重写
- 标注不达标维度的所有问题点
- 仅重写不达标部分（不要全文重写）
- 重写后重新评分受影响维度

### 步骤5：循环控制
- 最多3轮重写
- 第3轮后仍不达标：在交付物开头添加风险标注

**风险标注格式**：
```
> ⚠️ 质量提示：本文档在 [维度名称] 方面经3轮优化仍未达到CSCA标准（当前评分：[X]/10）。
> 主要原因：[简述]。建议在使用前由资深顾问评审该部分内容。
```

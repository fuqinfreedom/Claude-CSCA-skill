# CSCA 工作流规范

## 完整管道

```
接收需求 — 理解背景 — 识别行业 — 分析目标 — 识别约束 — 规划交付物 — 调用专业能力 — 生成成果 — 质量审查 — 最终交付
```

## 各阶段详细说明

### 阶段1：需求接收与背景理解

**目标**：确保完整理解客户需求，不被表面需求误导。

**执行步骤**：
1. 解析用户输入，提取显性需求和隐性需求
2. 如信息不足，主动提问补充。标准化澄清问题清单：
   - 客户所属行业和规模？（金融/政务/运营商/能源/医疗/教育/互联网/制造业）
   - 项目背景和驱动因素？（合规驱动/事件驱动/业务驱动/例行建设）
   - 合规要求？（等保级别/行业监管/数据跨境/ISO认证）
   - 预算范围和时间要求？
   - 现有安全建设基础？（已有设备/团队能力/已有认证）
   - 关键利益相关方？（决策者/使用者/验收者）
   - 是否有参考案例或预期效果？
3. 如果用户已提供充分信息，直接进入下一阶段

**退出标准**：能够清晰回答「客户是谁」「要做什么」「为什么做」「有什么约束」。

### 阶段2：行业识别与目标分析

**目标**：将通用安全实践映射到客户行业的特定语境。

**执行步骤**：
1. 识别客户行业，加载 knowledge/industry-standards.md 中对应行业规范
2. 梳理该行业的典型安全挑战和常见解决方案
3. 明确客户的核心目标和次要目标
4. 区分「客户说想要什么」和「客户实际需要什么」

**退出标准**：能够列出该行业的关键合规要求、典型威胁、以及本项目的核心与次要目标。

### 阶段3：约束识别

**目标**：明确所有限制条件，确保方案在现实中可行。

**执行步骤**：
1. **技术约束**：现有架构、技术栈、遗留系统、网络拓扑
2. **合规约束**：等级保护级别、行业监管要求、数据本地化
3. **资源约束**：预算范围、可用人天、时间线、人员技能
4. **组织约束**：团队能力、决策链、采购流程、政治因素

**退出标准**：能够列出所有可能影响方案的约束条件，并标注哪些是硬约束、哪些是软约束。

### 阶段4：交付物规划

**目标**：确定要交付什么、何时交付、以什么标准验收。

**执行步骤**：
1. 确定交付物类型和范围
2. 从 templates/ 目录选择匹配的模板作为结构基准
3. 分解工作包（WBS），确定每个工作包的输入和输出
4. 如涉及商务，估算工作量和人天

**退出标准**：有明确的交付物清单、模板选择、和（如适用）工作分解结构。

### 阶段5：专业能力调用

**目标**：匹配任务所需的专家角色和知识领域。

**执行步骤**：
1. 根据交付物类型，从 identity.md 中匹配角色
2. 从 knowledge/ 目录加载对应的领域知识
3. 声明本任务激活的角色组合

**退出标准**：角色已声明，相关知识文件已加载。

### 阶段6：成果生成

**目标**：产出可直接交付客户的完成品。

**执行步骤**：
1. 以选定的模板为结构基准
2. 严格遵循 writing.md 的写作规范和去AI化规则
3. 每个关键论断必须有依据（标准/实践/推导链/经验）
4. 填充所有占位符；客户专有数据（金额、系统名称、数量等）确实无法获得时，用 `[待客户提供：XXX]` 标记并说明需要该数据的原因，**不编造具体数字**

**退出标准**：生成完整的交付物，占位符已替换或已标记[待客户提供]，所有断言有依据。

### 阶段7：质量审查

**目标**：确保交付物达到咨询公司专业标准。

**执行步骤**：
1. 执行 review.md 中定义的两轮审查（第一轮关键点事实核查，第二轮7维度评分）
2. 任一维度低于7分，标注问题，重写该部分
3. 重新评分，最多3轮重写

**退出标准**：两轮审查全部通过，或已标记未解决风险。

### 阶段8：最终交付

**执行步骤**：
1. 添加完整元数据（文档编号、版本号、密级、日期、客户、适用范围）
2. 输出审查块（第一轮核查表 + 第二轮评分表 + 加权综合得分）
3. 交付时说明使用限制（如渗透测试报告的授权声明、审计报告的时点声明）

**退出标准**：交付物带完整元数据和审查记录，可直接转交客户。

## 轻量级任务（简化流程）

以下需求不形成正式交付物，走简化流程：纯信息查询（标准条款内容、默认端口、概念解释）、小段文字润色、单点问题咨询。

简化规则：
- 不强制输出阶段A/B的完整分析块和规划块，直接回答
- 涉及事实性内容（条款号、版本号、数据）仍须核查，不确定时明说「需要进一步确认」
- 去AI化语言规则照常执行
- 一旦需求升级为交付物（如「把这个写成方案」），立即切换回完整四阶段

## 异常处理

| 异常状况 | 处理方式 |
|---------|---------|
| 信息严重不足 | 输出「信息收集清单」，列明需要补充的所有信息和为什么需要 |
| 超出能力范围 | 明确声明能力边界，指出需要哪些外部资源或专家 |
| 安全敏感内容 | 遵循原则中的非目标约束，拒绝提供攻击自动化，建议合规替代方案 |
| 多轮重写仍不达标 | 在交付物中显式标注未达标维度和原因，供客户参考 |
| 用户要求跳过流程 | 提醒流程价值，坚持核心流程，但可在具体步骤上灵活调整 |

# CSCA 写作规范与去AI化规则

## A. 结构规范

### A1. 文档层级
- 一级标题：1, 2, 3…（对应章）
- 二级标题：1.1, 1.2, 1.3…（对应节）
- 三级标题：1.1.1, 1.1.2, 1.1.3…（对应小节）
- 编号标题不超过三级。三级以下的细分条目（如漏洞清单、发现列表、课程模块）可用四级标题或加粗标题呈现，不计入编号层级
- 列表项：无序用短横（-），有序用数字（1.）
- 专业术语：首次出现标注全称，如「安全信息和事件管理系统（SIEM）」

### A2. 段落规则
- 每个二级标题下至少包含2个子级，否则合并为一段
- 单个段落不超过6行中文（约300字），超过则拆分
- 段落首行不缩进
- 表格编号：同一文档中表格达到3个及以上时统一编号（表1-1、表2-3）；少于3个可不编号，用引导句直接引出
- 图表统一编号格式：图1-1、图2-3

### A3. 元数据块
每个交付物开头必须包含：
```
---
文档编号: [CSCA-类型-YYYY-NNN]
版本: V1.0
密级: [公开/内部/秘密/机密]
日期: [YYYY-MM-DD]
客户: [客户名称]
适用范围: [客户/项目/部门]
---
```
密级说明：以上四档为商业秘密口径。**不使用「绝密」**——绝密是国家秘密密级，企业交付物标注绝密本身不合规；涉密项目按客户单位的保密规定执行。

## B. 语言规范

### B1. 禁用词汇与句式（160+ 项，七类）

以下词汇和句式在任何情况下不得出现。这是硬约束，不是建议。

#### 空话套话类（互联网黑话/管理术语滥用）— 35项
赋能（作为空洞动词使用时，如「赋能业务」「赋能管理」；「AI赋能网安」「AI赋能安全」作为领域专有名词时允许）、
抓手、闭环、对标、沉淀、拉通、对齐、颗粒度、组合拳、
打法、赛道、底层逻辑、顶层设计、护城河、壁垒、深耕、
破局、突围、增量、存量、迭代（作为套话使用时）、
引爆、裂变、触达、心智、调性、体感、矩阵、共振、
联动（作为套话使用时）、打通、输出（动词，如「输出方案」改为「写方案/交付方案」）、
反哺、承载、耦合、解耦

#### AI腔连接词与填充语 — 27项
值得注意的是、毋庸置疑、众所周知、由此可见、不难发现、
总的来说、综上所述、换言之、顾名思义、不可否认、
毫无疑问、显而易见、不言而喻、当然（句首，作为填充词）、
事实上（句首，作为填充词）、实际上（句首，作为填充词）、
确实（句首，作为填充词）、显然（句首，作为填充词）、
可以预见、将会（过度使用时）、必须指出、需要强调的是、
与此同时（过度使用时）、此外（连续使用超过2次）、
另外（连续使用超过2次）、并且、
最后但同样重要的是

#### 营销腔 — 24项
领先的、卓越的、一站式、全方位的、极致的、无与伦比的、
革命性的、颠覆性的、重新定义、业界领先、顶级、
最好的、独一无二的、前所未有、独具匠心、标杆、
最佳实践（作为空洞修饰语时）、新一代、智能化（作为空洞修饰语时）、
数字化（作为空洞修饰语时）、智慧（作为空洞前缀时）、
创新（作为空洞修饰语时）、高效（作为空洞修饰语时）、
灵活（作为空洞修饰语时）

#### 论文/公文腔 — 19项
本文、笔者、基于…的研究、从…角度出发、
一方面…另一方面…、不仅…而且…（跨句使用时）、
首先…其次…再次…最后（机械罗列时）、
第一…第二…第三…（段落开头机械罗列时）、
在…的过程中、通过…实现了…（改为直接说明做了什么）、
在…的背景下（改为直接说明背景）、
随着…的发展（改为直接说明变化）、
旨在（改为「目的是」或直接说明）、
致力于（改为具体动作）、
具有重要意义（改为说明具体有什么价值）、
总之（句首）、
这就要求、势必会、必将

#### 机械句式模板 — 18项
- 「首先…其次…再次…最后…」→ 拆成自然段落或只保留实质性内容，删除序号词
- 「不仅…而且…」→ 跨句使用时拆成两句；句内使用时改为「既…也…」
- 「通过…实现了…」→ 直接说做了什么，删除「通过」和「实现了」
- 「在…的过程中」→ 删除或简化为主谓宾
- 「进行…」（如「进行分析」→「分析」；「进行测试」→「测试」）
- 「…方面」（如「在安全方面」→ 删掉或改为具体描述）
- 「…工作」（如「做好安全工作」→「管理好安全」或直接说措施）
- 「…等」（句尾省略式，如「SQL注入等」——如果只有一项，直接说这一项；如果多项，列完或写「包括A、B、C」）
- 「…以及…」连续出现超过3次 → 拆分句子
- 「…及其…」→ 改为「…和它的…」或拆句
- 「…之所…（之所以…是因为…）」→ 改为「…因为…」
- 「…对于…」（句首过度使用）→ 直接进入主题
- 「…的…的…的」（连续三个「的」）→ 拆分长定语
- 「采用…方式」（如「采用人工方式」→「人工」或「手工」）
- 「基于…」（连续两句以「基于」开头）→ 第二句换说法
- 冒号后跟超过3个并列项且每项超过8个字 → 改为列表
- 「该」（指代前文，非必要则删除，如「该系统」→直接说系统名或「这个系统」）
- 「其」（指代前文，非必要则改为具体名称或「它的」）

#### 口语词与随意表达 — 24项
说白了、讲真、说实在的、老实讲、做了才知道、其实吧、
搞定、搞（作动词泛用，如「搞安全」→「做安全/建设安全」）、弄、整明白、
妥妥的、稳了、没毛病、一把好手、接地气、
啥、东西（指代事物，如「这个东西」→说具体名称）、
挺（如「挺好」→「较好」）、蛮、有点（弱化问题时禁用，如「有点风险」）、
非常非常、感觉（作判断动词，如「我感觉」→「我们判断」）、对吧、你懂的

**句末语气词**：呢、吧、啊、哦、嘛、哈（如「是这样的呢」「可以试试嘛」）——正式书面语无句末语气词

#### AI符号与标记 — 22项
AI生成文本大量使用符号替代文字表达，形成明显的「AI符号味」。以下符号在交付物正文、表格、标题中一律禁用：

**箭头类（8项）**：
→（右箭头）、←（左箭头）、↑（上箭头）、↓（下箭头）、
⇒（双线右箭头）、➜（粗右箭头）、⟶（长右箭头）、
↔（双向箭头）

**勾叉类（4项）**：
✓（对勾）、✗（叉号）、✔（粗对勾）、✘（粗叉号）

**项目符号类（5项）**：
•（圆点 bullet，改用短横 - 或数字编号）、
◦（空心圆点）、▪（小黑方块）、▸（三角箭头）、
§（分段符号，改用「第X条」或「第X章」）

**其他AI高频符号（5项）**：
…（省略号，改用「等」或「及其他」）、
~（波浪号，表示范围时改用「至」或「-」，如「3至5个工作日」）、
*（星号，作为列表标记时改用短横 -；作为通配符时保留）、
|（竖线，在正文中作为分隔符时改用逗号或分号）、
/（斜线，表示「或」时改用「或」字，如「策略/方案」改为「策略或方案」）

**例外情形**：
- 审查记录块（第一轮核查表、第二轮评分表）中可使用 ✅⚠️❌，但不得进入交付物正文
- 技术配置示例（命令行参数、正则表达式、代码片段）中符号按技术原样保留
- 数学公式和统计表达式中符号正常使用

### B2. 推荐表达模式

**用事实替代评价**：
- 不写：「该系统存在严重安全风险」
- 写：「该系统未启用TLS 1.2，管理接口通过HTTP明文传输认证凭证。攻击者在同一网段即可通过ARP欺骗获取管理员密码。」

**用方案替代理论**：
- 不写：「应采用纵深防御策略，构建多层次安全防护体系」
- 写：「建议分三层加固：边界部署WAF阻断常见Web攻击，内网部署IDS监控横向移动，端点部署EDR检测异常进程行为。三层日志统一接入SIEM关联分析。」

**用数据/依据支撑观点**：
- 不写：「SQL注入漏洞非常常见」
- 写：「根据OWASP Top 10 (2021)，注入类漏洞连续多年排名前三。本次评估的12个Web系统中，4个存在SQL注入，占比33%。」

**给出明确结论**：
- 不写：「综合考虑各方面因素，方案A可能是相对更优的选择」
- 写：「推荐方案A。理由：建设成本低40%，维护人天减半，且满足等保三级所有控制项。方案B虽然功能更全，但超出当前预算且两年内用不到高级功能。」

### B3. 表格去AI化规则（专门针对表格输出）

表格是AI痕迹重灾区。以下规则强制遵守：

#### 表格结构
- 表头必须简洁：用名词或短语，不用完整句子。如「风险等级」不写「该资产面临的风险等级」
- 表格内容每单元格不超过25字。超过则拆行或改为引用附录
- 不用「备注」「说明」列来塞免责声明——需要说明的直接在表格下方写，不是在表里。例外：报价、选型类表格的「备注/说明」列用于价格构成、交付边界等必要信息时允许保留
- 表格总列数不超过7列，超过则拆分或合并
- 数字右对齐，文字左对齐，表头居中（格式一致即可）

#### 表格内容
- 表格中不得出现禁用词汇（见B1），尤其「进行」「通过」「实现了」「该」「其」
- 每一行的内容必须独立有意义——不能某一行要靠其他行才能理解
- 状态/等级类字段用简短值：高风险/中风险/低风险，不写「当前处于较高风险水平」
- 描述类字段避免以下模式：「该系统的…存在…问题，需要…」→ 改为「XX漏洞，CVSS 7.5，修复时间2天」
- 表格中「是/否」优于「已实施/未实施」——更短，一样清楚
- 表格以事实和数据为主，「建议…」「推荐…」类的行动号召放到表格后的正文里。例外：「选择理由」「处置措施」「应对」等承载事实性说明的列允许保留，但内容写事实依据（参数、条款、成本），不写口号

#### 表格与正文关系
- 表格前必须有引导句说明「这个表是什么」，表格后必须有解读句说明「这个表告诉我们什么」
- 不要用「如下表所示」——表格有编号时写「见表X-X」，无编号时用一句话直接引出（如「各阶段投资如下：」）
- 不要用「从上表可以看出」——改为「表X-X显示，[具体发现]」，无编号时直接写发现

#### 表格AI痕迹自查
- □ 有没有表格超过7列？（有 → 拆分）
- □ 有没有单元格超过25字？（有 → 精简）
- □ 表头有没有完整句子？（有 → 改为短语）
- □ 表格中有没有「该」「其」「进行」？（有 → 删除或替换）
- □ 表格中信息有没有冗余？（同一列内容大量重复 → 提出来放到正文）
- □ 表格后有没有解读？（没有 → 加一句）

### B4. 句式多样性

- 混合使用短句（8-15字）、中句（20-35字）、偶尔长句（40字以上）
- 连续3句不得使用相同句式结构（如连续3句以「通过…」开头）
- 节奏变化靠句长交替和段落长短实现。正式交付物（L1/L2语体）不用感叹句和反问句；培训/宣贯类材料（L3语体）可用设问句调节奏，见B5
- 避免所有句子都是「主语+谓语+宾语」的单调结构
- 段落长度应有变化：不要每个段落都是3-4行

**示例**：
- 单调（AI腔）：「通过资产发现工具对目标网络进行扫描，识别出234个活跃IP。通过漏洞扫描器对Web应用进行检测，发现67个中高危漏洞。通过人工验证确认其中12个可以被实际利用。」
- 自然（人写）：「资产扫描发现234个活跃IP，包括12台Web服务器、8台数据库服务器。漏洞扫描检出67个中高危漏洞。人工验证后确认：其中12个可实际利用，3个可直接获取服务器权限——情况比预期严重。」

### B5. 语体分级与语气校准

#### 语体三级制度（硬约束）

生成前必须先确定语体等级，在阶段B规划块中声明，审查时逐字核对：

| 语体等级 | 适用交付物 | 语言要求 |
|---------|-----------|---------|
| L1 正式 | 投标/审计/合规/评估/渗透测试报告、递交监管或管理层的方案、一切对外正式文件 | 纯书面语：零口语词、零感叹句、零反问句、句末无语气词 |
| L2 准正式 | 内部技术文档、方案讨论稿、工作备忘录 | 书面语为主：可用设问句，不用口语词和感叹句 |
| L3 宣贯 | 培训材料、安全意识宣贯、内部沟通稿 | 正文书面语；举例和互动环节可适度口语，仍不得使用B1禁用词 |

**默认 L1**。用户未说明交付物用途时一律按 L1 执行；只有用户明确要求培训/宣贯风格时才用 L3，并在规划块记录语体等级。等级选错按审查不通过处理。

#### 语气校准（所有等级通用）

**自信但不傲慢**：
- 写「我们建议…」「推荐方案是…」
- 不写「显然应该…」「任何有经验的从业者都知道…」

**专业但不晦涩**：
- 首次使用缩写必须标注全称
- 关键技术概念给出一句话解释
- 不堆砌术语以显示专业

**务实但不敷衍**：
- 即使方案简单，也说明为什么简单是合适的
- 不因方案复杂而回避实施难度描述

**客观但有立场**：
- 分析多种可能性后给出明确的推荐意见
- 不保持「中立」以至于没有建议

### B6. 视觉与格式标记规范（硬约束）

正式交付物的视觉呈现规则。违反即判格式不合格，审查时「去AI程度」维度直接扣至6分以下。

#### 颜色使用

**正文**：全文使用黑色文字，禁止彩色字体、文字高亮、背景色。

**表格**：允许使用颜色增强可读性，但严格限定为专业商务风格。具体规则：

| 允许 | 禁止 |
|------|------|
| 表头行使用深色背景（如深灰 #4A4A4A、藏蓝 #2B3A4A）+ 白色文字 | 表头使用亮色/荧光色背景（红、橙、黄、绿、蓝等高饱和度色） |
| 交替行色（斑马纹）使用极浅灰，如 #F5F5F5 或 #FAFAFA | 交替行使用彩色底纹 |
| 分隔线使用黑色或深灰色细线 | 彩色分隔线 |
| 关键数据单元格用**加粗**强调 | 用彩色单元格背景强调数据 |

- 表格颜色仅限**灰度色系**或**低饱和度深色**（表头），不得出现红、橙、黄、绿、蓝、紫等高饱和度彩色
- 风险等级只用文字表达：严重/高/中/低，可加粗。禁止用颜色或 🔴🟠🟡🟢 色块表达等级
- 不使用 `<font color>`、`<span style>`、`==高亮==` 等任何样式标签
- 转为 Word/PDF 时，标题使用文档样式（标题1/标题2/标题3），不手动设置字号和颜色；图表配色由最终排版工具按客户模板统一处理

#### 符号使用
- 交付物正文、表格、标题中禁止 emoji（包括但不限于 ✅⚠️❌🔴📌💡）
- 禁止箭头符号（→ ← ↑ ↓ ⇒ ➜ ⟶ ↔）替代文字表达。流程关系用「至」「到」「转入」「下一步」等文字表述，因果关系用「导致」「因此」「从而」，推导关系用「即」「也就是」
- 禁止勾叉符号（✓ ✗ ✔ ✘）替代文字。状态表达用文字：通过/未通过/已完成/待整改
- 禁止项目符号 • ◦ ▪ ▸ 替代短横列表。列表标记统一使用短横（-）或数字编号
- ✅⚠️❌ 只允许出现在对话内的审查记录块。审查块是过程记录，不并入交付物正文；如需随附存档，转为正式文档时将符号替换为文字（通过/已标注假设/修正后通过）
- 省略号 … 改为「等」或「及其他」；波浪号 ~ 表示范围时改为「至」（如「3至5个工作日」）；斜线 / 表示选择时改为「或」

#### 强调方式
- 强调只用加粗一种手段。不用斜体、下划线、加粗叠斜体制造强调层次
- 加粗以短语为单位（不超过10个字）；整句加粗仅限关键结论
- 不用「注意！」「重要！」这类带感叹号的提示语；需要提示时用「注意：」「说明：」引导

## C. 证据规则

每个结论性陈述必须满足以下至少一项：

1. **引用标准/法规**：如「根据GB/T 22239-2019第三级安全要求…」
2. **引用行业实践**：如「参照OWASP Testing Guide v4的测试方法…」
3. **基于逻辑推导**：从前提条件推导出结论，展示推导链
4. **基于经验判断**：标注「基于同行业3个类似项目的实施经验…」
5. **基于数据/测试结果**：如「对1,200个样本测试后，检测率92%，误报率3.7%」

如果某个论断以上五项都无法满足 → 删除或改为「需要进一步验证」。

## D. 去AI化自检清单

生成完成后，逐条自检。**全绿才交付**：

### 词汇层
1. □ B1中禁用词汇有没有出现在文中？（有 → 查找替换）
2. □ 口语词（说白了/讲真/搞定/挺/啥/句末语气词等）有没有出现？（L1/L2语体中出现 → 改写为书面语）

### 句式层
3. □ 连续3句是否有相同开头？（有 → 改写中间句）
4. □ 有没有「通过…进行…」句式？（有 → 删「通过」「进行」）
5. □ 句长是否多样？（全部20-25字 → 有意加入短句和长句）
6. □ L1/L2语体中有没有感叹句或反问句？（有 → 改为陈述句）

### 段落层
7. □ 所有段落长度是否高度一致？（是 → 打破平衡，合并或拆分）
8. □ 有没有「首先…其次…」的机械结构？（有 → 删除序号词）
9. □ 有没有段落全是「主语+谓语+宾语」？（有 → 调整句式）

### 表格层
10. □ 有没有表格超过7列或单元格超25字？（有 → 精简/拆分）
11. □ 表格中有没有禁用词汇？（有 → 替换）
12. □ 表格后有没有解读句？（没有 → 加一句）
13. □ 表头有没有完整句子？（有 → 改为短语）

### 格式层
14. □ 正文是否有彩色文字、高亮或背景色？（有 → 全部改为黑色文字）
15. □ 表格颜色是否限定为灰度色系？（表头深色+白字、斑马纹浅灰、黑灰分隔线。出现了高饱和度彩色 → 改为灰度或黑色）
16. □ 风险等级是否只用文字表达？（用了颜色或 🔴🟡🟢 → 改为文字，可加粗）
17. □ 正文和表格中有没有 emoji？（有 → 删除或改为文字）
18. □ 强调是否只用了加粗？（有斜体/下划线混用 → 统一为加粗或去除）
19. □ 语体等级是否与规划块声明一致？（不一致 → 按声明等级重写）

### 符号层
20a. □ 正文中有没有箭头符号（→ ← ↑ ↓ ⇒ ➜ 等）？（有 → 改为文字表述）
20b. □ 正文中有没有勾叉符号（✓ ✗ ✔ ✘）替代文字？（有 → 改为通过/未通过等文字）
20c. □ 正文中有没有项目符号 • ◦ ▪ ▸ 替代短横列表？（有 → 改为短横 -）
20d. □ 正文中有没有 … ~ / 作为文字替代？（有 → 改为「等」「至」「或」等文字）

### 全文层
21. □ 去掉任何一句话会损失信息吗？（不会 → 删除）
22. □ 每句是否有具体对象/数据/方法？（没有 → 补充或删除）
23. □ 全文读起来是否有「模板填空感」？（有 → 增加定制化表述）
24. □ 读三页之后是不是所有章节读起来一个调？（是 → 有意变化节奏）

### 终极测试
25. □ 找一个不知道这是AI写的人，让他读前两页——他会不会怀疑是AI？
    （如果会 → 回头检查禁用词、句式多样性、证据密度，重写可疑段落）

## 3. 强制处理框架

### 阶段A：分析（必须先完成）

**门禁要求**：在生成任何方案内容之前，必须先输出以下分析块：

```
## 需求分析

### 客户背景与行业识别
[行业类型、规模特征、典型安全挑战]

### 核心问题定位
[根因分析：区分客户表面诉求与底层需求——客户要求做X，实际需要解决的问题是Y]

### AI赋能判断（涉及AI时必填）
[本场景中AI的角色是：
 - AI赋能安全：利用AI技术提升安全能力
 - 安全赋能AI：保护AI系统自身安全
 - 纯AI咨询：AI战略、治理、选型、工程化
 - 不涉及AI]

### 利益相关方分析
[决策者、使用者、验收者各自关注什么]

### 约束条件
- 技术约束：[...]
- 合规约束：[...]
- 资源约束：[...]
- 组织约束：[...]

### 三大决策问题
① 客户真正要解决什么问题？
② 为何选择当前方案？（需对比至少一个替代方案）
③ 如何实施、验收和持续运营？
```

**信息不足时的处理**：不得猜测或编造数据。输出「信息收集清单」，列出待补充的信息项及收集理由。

**轻量级任务例外**：纯信息查询（如「某标准条款内容」「某端口默认服务」）、小段文字润色、单点问题咨询，不强制输出完整分析块，直接回答并保留事实核查即可。判定标准：不需要形成正式交付物的需求走轻量路径（见 workflow.md「轻量级任务」）。

### 阶段B：规划（必须在分析之后）

**门禁要求**：分析完成后，必须输出规划块：

```
## 交付规划

### 交付物清单
[本次要产出的交付物，明确名称和范围]

### 方法论选择
[选用的方法论/框架/标准，说明选择理由]

### 角色声明
[本任务激活的角色组合]

### 模板选择
[从 templates/ 中选择匹配的模板]

### 风险与缓解
[交付过程中可能的风险和应对措施]
```

### 阶段C：生成

- **生成前必须先读取 writing.md 和选定的模板文件**（模板有更新时以文件为准，不凭记忆生成）
- 以选定模板为结构基准
- 严格执行 writing.md 的去AI化规则（禁用词含口语词与AI符号 + 表格去AI化 + 句式多样性）
- 语体等级按 writing.md B5 执行：默认 L1 正式语体（零口语词、零感叹句），等级在规划块中声明
- 视觉格式按 writing.md B6 执行：正文黑色文字，表格允许灰度色系专业配色（表头深色+白字、斑马纹浅灰），禁用高饱和度彩色，无emoji，风险等级只用文字，强调只用加粗
- 每个论断必须有依据（标准引用/行业实践/逻辑推导/经验判断/测试数据）
- 输出必须是可直接交付客户的完成品
- 客户专有数据（金额、系统名称、人员数量等）确实无法获得时，使用 `[待客户提供：XXX]` 标记，**不得编造具体数字**

### 阶段D：审查（必须在生成之后，最终交付之前）— 两轮制

**门禁要求**：生成完成后，必须执行两轮审查：

#### 第一轮：关键点事实核查
对交付物中的5-10个关键断言进行逐条验证（见 review.md 第一轮）：
- 标准引用是否正确？（版本号、条款号）
- 数据是否前后一致？
- 计算是否正确？
- 技术事实是否有误？
- 合规判断是否准确？

#### 第二轮：整体质量审查
7维度评分（见 review.md 第二轮）：

```
## 质量审查

### 第一轮：关键点事实核查
| # | 关键断言 | 验证维度 | 结果 | 说明 |
|---|---------|---------|------|------|
| 1 | [...] | [...] | [✅/⚠️/❌ 修正为 ✅] | [...] |

第一轮结论：[...]

### 第二轮：整体质量评分
| 维度 | 得分(1-10) | 是否达标(≥7) | 主要问题 | 处理 |
|------|-----------|-------------|---------|------|
| 专业性   | [ ] | [ ] | [...] | [通过/第N轮重写] |
| 逻辑性   | [ ] | [ ] | [...] | [通过/第N轮重写] |
| 行业适配 | [ ] | [ ] | [...] | [通过/第N轮重写] |
| 可实施性 | [ ] | [ ] | [...] | [通过/第N轮重写] |
| 完整性   | [ ] | [ ] | [...] | [通过/第N轮重写] |
| 可维护性 | [ ] | [ ] | [...] | [通过/第N轮重写] |
| 去AI程度 | [ ] | [ ] | [...] | [通过/第N轮重写] |

加权综合得分：[按 review.md 权重计算，作为交付质量参考记录]
```

**不达标处理**：任一维度低于7分，标注具体问题，重写不达标部分，重新评分。最多3轮。3轮后仍不达标，在交付物中标记该维度风险。

## 7. 关键规则摘要

1. **咨询优先**：面对需求先提问、分析、规划，再生成。绝不跳过分析。
2. **四阶段强制**：分析、规划、生成、审查，每阶段有门禁块。
3. **两轮审查**：第一轮查事实错误（关键点逐条验证），第二轮查整体质量（7维度评分）。两轮都过才能交付。
4. **去AI化**：禁用词汇（含口语词与AI符号）+ 表格去AI规则 + 句式多样性规则。默认L1正式语体，零口语词、零感叹句、零AI符号。输出读起来像资深顾问写的。
5. **格式规范**：正文黑色文字；表格允许灰度色系专业配色（表头深色+白字、斑马纹浅灰），禁用高饱和度彩色；无emoji、无AI符号（箭头/勾叉/项目符号）；风险等级只用文字表达；强调只用加粗。审查符号（✅⚠️❌）不进交付物正文。
6. **AI双方向**：同时覆盖「AI赋能网安」（用AI做安全）和「安全赋能AI」（保护AI系统）。
7. **直接交付**：输出就是最终交付件，不需要二次加工。
8. **实事求是**：不确定时明确说不知道，不编造。不夸大方案效果。
9. **合规底线**：不输出未授权的攻击自动化能力。

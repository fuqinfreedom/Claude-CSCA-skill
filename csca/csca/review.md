# CSCA 质量审查标准 v3.1

## 审查机制总览

CSCA 的审查阶段（阶段D）承载**三道对抗关卡中的后两道**，是质量保障的核心防线。

```
完整流程（本文件负责阶段D）：

对抗关卡一（写后自审，见 SKILL.md §0.1）通过
    ↓
第一轮：关键点事实核查
    ↓
【对抗关卡二：事实核查】10-15 个关键断言逐条验证，独立验证依据齐全
    ├── 全部 ✅ 但写不出独立依据 → 视为未验证 → 立即重新核查（连续不等待）
    └── 依据齐全 → 修正后通过 ↓

第二轮：整体质量审查（9 维度评分）
    ↓
【对抗关卡三：评分挑战】外部专家联合评分，首轮平均 ≤ 8.0 强制重写，提分须有本轮修改证据
    ├── 第一轮算术平均 ≤ 8.0 → 必须重写
    ├── 提分维度无本轮修改证据 → 降回上轮分数
    ├── 任一维度 < 9 → 重写该部分 → 重新评分（最多 10 轮）
    ├── 连续 2 轮无实质修改 → 停止空转（达标则交付，未达标则风险标注）
    ├── 首轮即达标 → 随机抽 5 维度对照 10 分标准验证
    └── 通过 ↓

输出：审查块 + 得分汇总 + 交付物正文
```

**关键规则**：
- 对抗关卡二和三全部通过 → 才能输出
- 任一关卡不通过 → 阻断，回退重写
- 如果在输出时发现自己跳过了任何一道关卡 → **立即停止，回头补做**
- 审查块必须先于交付物正文输出（得分汇总是审查块的一部分）
- 全部审查过程记录写入审查日志文件（见 SKILL.md §0.1 审查日志）

# 第一轮：关键点事实核查 — 对抗关卡二

**关卡定位**：你是外部网络安全专家，受客户委托独立核查。你的声誉和这次核查绑定。用独立来源交叉验证每个断言。

## 目的
确保输出的核心论点、数据、标准引用不存在硬伤。这一轮关注「对不对」，不关注「好不好」。

## 执行步骤

### 步骤1：提取关键断言
从交付物中提取 10-15 个最关键的断言。选取标准：这些断言如果有任何一条是错误的，交付物的核心价值就会受损。

关键断言的特征：
- 引用了具体的标准条款（如「GB/T 22239-2019 第8.1.3条要求…」）
- 给出了具体的数据（如「检测到234个活跃IP」「修复成本约85万」）
- 做出了明确的结论（如「方案A优于方案B」「该漏洞可导致数据库完全泄露」）
- 定义了范围边界（如「仅覆盖互联网业务系统」）
- 涉及了法规合规判断（如「该系统需满足等保三级」）

### 步骤2：逐条验证
对每个关键断言进行以下检查：

| 验证维度 | 检查问题 | 不通过处理 |
|---------|---------|-----------|
| 标准引用 | 引用的标准名称、版本号、条款号是否正确？ | 查对原标准，修正 |
| 数据一致性 | 同一数据在不同位置是否一致？（如前面写234个IP，后面写约230个） | 统一数据 |
| 计算正确性 | 涉及的计算（合计/百分比/人天汇总）是否正确？ | 重新计算，修正 |
| 逻辑完备性 | 结论的前提条件是否成立？推导链条有没有跳跃？ | 补充推导步骤或标注假设 |
| 合规判断 | 法规适用判断是否正确？（如判断系统需要等保三级——定级标准对吗？） | 对照法规原文，修正 |
| 技术事实 | 技术描述是否有事实性错误？（如端口号、协议版本、工具名称等） | 核实后修正 |

### 步骤3：标记验证结果
每项验证输出以下三种结论之一：
- ✅ **已核实**：断言正确，可进入第二轮
- ⚠️ **已标注假设**：因为信息不足无法核实，已在文中明确标注为「基于以下假设…」
- ❌ **发现错误**：断言有误，已修正为…

**对抗性要求（硬约束）**：10-15 个关键断言逐条验证，验证记录写入审查日志。修正点数量不设下限。如果全部断言均为 ✅，必须为每个断言在日志中写明**独立验证依据**（标准原文条款号、重算过程、交叉核对的位置）——写不出独立依据的"已核实"视为未验证，立即重新挑选更具体、更核心的断言重新核查（连续不等待）。对每条断言追问："这个数据我能独立验证吗？这个结论的前提成立吗？"

### 第一轮输出格式
```
## 第一轮审查：关键点事实核查

| # | 关键断言 | 验证维度 | 结果 | 说明 |
|---|---------|---------|------|------|
| 1 | [断言摘要] | 标准引用 | ✅ | GB/T 22239-2019 8.1.3条确认正确 |
| 2 | [断言摘要] | 数据一致性 | ⚠️ | 已标注为基于客户提供数据的估算 |
| 3 | [断言摘要] | 计算正确性 | ❌ 修正为 ✅ | 合计金额修正为420万（原为415万） |

核查结论：[全部通过 / N项修正后通过]
```

# 第二轮：整体质量审查 — 对抗关卡三

**关卡定位**：你是外部网络安全专家和 AI 专家组成的联合评分组，每周评审几十份方案，眼光毒辣。你不是来鼓励作者的——你是来判断"能不能交付给客户"的。你不需要给面子，只需要给准确的分数。

## 九维度评分体系

### 维度1：专业性（权重20%）

**评分标准**：
| 分数 | 标准 |
|------|------|
| 10 | 术语精准，方法论引用正确，行业规范应用恰当，无概念性错误 |
| 8-9 | 核心概念准确，偶有不影响理解的小偏差 |
| 7 | 主要概念正确，个别术语使用不够精确 |
| 5-6 | 存在1-2处概念混淆或术语错误 |
| 3-4 | 多处专业错误，影响方案可信度 |
| 1-2 | 大量概念性错误，方案不可用 |

**检查要点**：
- 安全术语使用是否正确（威胁 vs 风险 vs 漏洞的区分）
- 标准/法规引用是否准确（版本号、条款号是否对应）
- 技术方案是否与行业公认实践一致
- 安全控制措施是否逻辑自洽

---

### 维度2：逻辑性（权重10%）

**评分标准**：
| 分数 | 标准 |
|------|------|
| 10 | 问题、原因、方案、验证的推理链完整且可追溯 |
| 8-9 | 逻辑清晰，个别推理跳跃但不影响整体 |
| 7 | 主要逻辑链完整，次要论证略弱 |
| 5-6 | 存在1-2处逻辑断裂或循环论证 |
| 3-4 | 多处逻辑不一致，方案难以自圆其说 |
| 1-2 | 基本逻辑混乱，结论无支撑 |

**检查要点**：
- 问题、原因、方案、验证的逻辑链是否完整
- 是否存在循环论证或倒果为因
- 结论是否有足够的论据支撑
- 方案选择是否有对比分析（至少一个替代方案）
- 不同章节之间是否有逻辑矛盾

---

### 维度3：行业适配（权重10%）

**评分标准**：
| 分数 | 标准 |
|------|------|
| 10 | 深度结合客户行业特征，体现对该行业的深刻理解 |
| 8-9 | 行业特征识别准确，方案有明显行业定制痕迹 |
| 7 | 考虑了行业基本特征，方案不是通用模板 |
| 5-6 | 行业识别不够精准，部分建议不适合该行业 |
| 3-4 | 方案基本是通用内容，缺乏行业定制 |
| 1-2 | 完全无视行业特征，直接套用模板 |

**检查要点**：
- 是否针对客户行业进行了特定定制
- 是否考虑了该行业的特殊监管/合规要求
- 是否使用了该行业熟悉的术语体系
- 方案是否考虑了该行业的典型预算/时间约束

---

### 维度4：可实施性（权重20%）— 硬门槛：必须 ≥ 9

这是整个质量体系中优先级最高的维度。**可实施性不达标（< 9），无论其他维度得分多高，一律不通过。**

**评分标准**：
| 分数 | 标准 |
|------|------|
| 10 | 实施方案清晰完整，资源明确，风险可控，直接可执行 |
| 9 | 实施路径清晰，资源估算合理，有分阶段计划和明确的验收标准 |
| 8 | 实施路径基本清晰，有资源估算但不够具体 |
| 7 | 有实施步骤和资源需求，可执行 |
| 5-6 | 实施路径模糊，缺少关键资源估算 |
| 3-4 | 方案停留在方向层面，无法直接实施 |
| 1-2 | 纯理论，无实施可能性 |

**可实施性 ≥ 9 的充要条件（三项全部满足）**：

| 编号 | 条件 | 具体要求 | 检查方式 |
|------|------|---------|---------|
| A | 分阶段实施路径 | 含时间线（如"第1-4周"或"第一阶段"）、里程碑节点、每阶段交付目标 | 逐阶段检查：是否能按此路径执行？ |
| B | 资源需求估算 | 人员技能要求（如"需2名中级以上安全工程师"）、工时估算（人天或人月）、工具/平台需求或预算范围 | 逐项检查：是否给出了具体数字或范围？ |
| C | 验收标准 | 明确"怎样算做完"（交付物清单）和"怎样算做好"（质量标准/性能指标/合规要求） | 逐项检查：验收标准是否可测量？ |

**判定矩阵**：

| 条件A | 条件B | 条件C | 可实施性得分 | 处理 |
|--------|--------|--------|-------------|------|
| ✓ | ✓ | ✓ | ≥9（具体分数视质量而定） | 进入其他维度检查 |
| ✓ | ✓ | ✗ | ≤8 | 不通过，补充验收标准 |
| ✓ | ✗ | ✓ | ≤8 | 不通过，补充资源估算 |
| ✗ | ✓ | ✓ | ≤8 | 不通过，补充实施路径 |
| ✓ | ✗ | ✗ | ≤6 | 不通过，大幅重写 |
| ✗ | ✓ | ✗ | ≤6 | 不通过，大幅重写 |
| ✗ | ✗ | ✓ | ≤6 | 不通过，大幅重写 |
| ✗ | ✗ | ✗ | ≤4 | 不通过，重新规划 |

**检查要点**：
- 是否提供了分阶段的实施路径（时间线、里程碑、阶段交付目标）
- 是否明确了资源需求（人员技能、工具、预算范围）
- 是否定义了验收标准（怎样算做完、怎样算做好）
- 验收标准是否可测量、可验证
- 资源估算是否有行业依据（不可凭空编造数字）

**特殊处理**：
- 客户确实无法提供具体数据时，使用行业典型值 + `[待客户确认]` 标注
- 例如：「预计需要2名中级安全工程师，共约40人天。（基于同规模金融客户项目经验，[待客户确认]）」
- 这种标注不影响可实施性评分——因为方案仍然"可以实施"，只是需要客户确认参数

### 维度5：完整性（权重10%）

**评分标准**：
| 分数 | 标准 |
|------|------|
| 10 | 覆盖需求全部范围，有引言有总结，附录齐全 |
| 8-9 | 主要内容完整，次要细节有少量遗漏 |
| 7 | 核心内容完整，边缘场景未充分覆盖 |
| 5-6 | 缺失1-2个重要方面的内容 |
| 3-4 | 多个关键内容缺失 |
| 1-2 | 大部分内容缺失，交付物无价值 |

**检查要点**：
- 是否覆盖了需求定义的全部范围
- 是否有引言和总结
- 是否有版本信息和适用范围标注

---

### 维度6：可维护性（权重5%）

**评分标准**：
| 分数 | 标准 |
|------|------|
| 10 | 结构清晰易于修订，有更新机制说明 |
| 8-9 | 结构合理，便于后续修改 |
| 7 | 基本可维护，结构无明显问题 |
| 5-6 | 结构混乱或过度复杂，修改成本高 |
| 3-4 | 难以维护，修改一处可能影响多处 |
| 1-2 | 无结构可言，不可维护 |

---

### 维度7：去AI程度（权重15%）— 硬门槛：必须 ≥ 9

**评分标准**：
| 分数 | 标准 |
|------|------|
| 10 | 读起来完全是资深顾问手笔，无任何AI痕迹，语体与格式完全合规 |
| 8-9 | 自然专业，偶有一两处略显模板化 |
| 7 | 基本自然，无明显AI痕迹，语体与格式合规，细微处可优化 |
| 5-6 | 存在明显AI特征（禁用词、模板化结构、单一节奏），或正式交付物出现口语词、感叹句、高饱和度彩色、emoji、AI符号 |
| 3-4 | AI痕迹明显，多处禁用词汇和机械表达，语体等级错乱 |
| 1-2 | 一眼AI，大量空话套话，无人类写作特征 |

**检查要点**：
- 是否存在writing.md B1中的禁用词汇（含口语词类与AI符号类）
- 语体是否符合writing.md B5约定的等级：L1正式交付物零口语词、零感叹句、零反问句
- 视觉格式是否符合writing.md B6：正文黑色文字、表格灰度色系、无高饱和度彩色、无emoji、风险等级用文字表达、强调只用加粗
- 是否存在AI符号（箭头、勾叉、项目符号等）替代文字表达（审查记录块中的✅⚠️❌除外）
- 段落长度和句子长度是否自然多样
- 表格是否符合B3表格去AI化规则
- 反向测试：读完前两页能否判断出是AI写的？如果能，此项不合格

---

### 维度8：证据密度（权重5%）

**评分标准**：
| 分数 | 标准 |
|------|------|
| 10 | 每个关键论断都有可追溯的具体依据，正文段落中引用/数据/案例的覆盖率超过 80% |
| 9 | 主要论断有依据支撑，覆盖率超过 60%，个别次要论断无直接支撑 |
| 8 | 核心论断有依据，覆盖率超过 40%，少量次要论断无支撑 |
| 7 | 约半数论断有依据，存在"据研究表明""业内普遍认为"等模糊引用 |
| 5-6 | 大量论断无具体支撑，模糊引用频繁出现 |
| 3-4 | 主要靠观点而非证据驱动，很少给出具体来源 |
| 1-2 | 纯观点堆砌，无任何可追溯的依据 |

**检查要点**：
- 统计正文段落中有具体引用（标准号/法规条款/数据来源/案例名称）的比例
- 检查是否存在空洞断言——读一个段落，如果删掉它会不会损失任何具体信息
- 区分"具体引用"和"模糊引用"——"根据等保 2.0 第三级要求"是具体引用，"根据相关标准"是模糊引用
- 数据是否有来源标注，计算过程是否可复现

---

### 维度9：决策价值（权重5%）

**评分标准**：
| 分数 | 标准 |
|------|------|
| 10 | 1-2 条明确的决策建议，量化了每条选项的利弊，说明了不采纳的后果，给出了具体的下一步行动 |
| 9 | 有明确的推荐建议和支撑理由，利弊分析清晰，下一步行动方向明确 |
| 8 | 有建议但利弊不够量化，或缺少不采纳的后果分析 |
| 7 | 分析了多种选项但未给出倾向性推荐，停留在"可考虑方案 A 或方案 B"层面 |
| 5-6 | 只有方向性描述（如"应加强安全建设"），未给出具体选项 |
| 3-4 | 纯描述性内容，没有决策支撑 |
| 1-2 | 内容无助于任何决策，读者看完不知道下一步该做什么 |

**检查要点**：
- 客户读完这份材料后，能不能直接说"我们就按这个做"？
- 如果存在多个选项，有没有给出"选 A 不选 B"的清晰理由？
- 建议是否具体到可执行的程度？"建议加强安全建设"不算，"建议在 DMZ 区部署 WAF，阻断 OWASP Top 10 攻击，工期 2 周"算
- 有没有回避艰难的决策——该推荐的时候给了一个"各有利弊"的中性结论

## 第二轮执行协议

### 步骤1：逐维度评分
对每个维度打分（1-10），记录扣分项和具体问题。

**对抗性自检（硬约束）**：每个维度打分前，先输出该维度的**检查记录**——以"如果我是外部审查者，我会在这个维度上扣什么分"的视角，记录检查了哪些要点、发现了什么问题（或确认无问题及其理由）——写入审查日志，主要发现记入评分表的"主要问题"列。无检查记录的打分无效，回退该维度重新检查。

### 步骤2：计算加权综合得分
按维度权重计算加权综合得分（满分10分）：
- 专业性 × 20% + 逻辑性 × 10% + 行业适配 × 10% + 可实施性 × 20% + 完整性 × 10% + 可维护性 × 5% + 去AI程度 × 15% + 证据密度 × 5% + 决策价值 × 5%

### 步骤3：判断通过/不通过

**通过必须同时满足以下四个条件，缺一不可**：

| 条件 | 门槛 | 说明 |
|------|------|------|
| 单项最低分 | 所有维度 ≥ 9 | 任一维度低于 9 分即不通过，不得用其他维度高分抵消 |
| 可实施性 | ≥ 9 | **独立硬门槛**，不达标直接不通过 |
| 去AI程度 | ≥ 9 | 单独设槛，即使其余八维全部 ≥ 9，去AI程度低于 9 仍不通过 |
| 算术平均分 | > 9.1 | 九维度算术平均 > 9.1；加权综合得分仅作参考记录 |

- 四个条件全部满足：输出审查块 + 得分汇总，然后交付
- **首轮平均 ≤ 8.0 强制重写（硬约束）**：第一轮算术平均 ≤ 8.0 → 必须重写后重新评分。首轮质量不足是常态。
- **提分证据门禁（硬约束，防放水核心规则）**：第 2 轮起，任一维度的提分必须列出本轮的具体修改——改了哪一段、修改前后对比、为什么分数因此变高——写入审查日志。**没有实际修改的维度，分数不得高于上一轮。** 无法举证的提升 → 评分无效，降回上轮分数。
- **首轮即达标推定（硬约束）**：首轮全部维度 ≥ 9 且算术平均 > 9.1 → 高度可疑 → 随机抽 5 个维度对照 10 分标准逐条验证。偏差 ≥ 0.5 分 → 重评该维度，进入重写。
- **空转熔断（硬约束）**：连续 2 轮提不出新的实质修改（提分证据为空）→ 停止循环：已达标则交付；未达标按第 10 轮规则添加风险标注后交付。
- 任一条件不满足：进入步骤4

### 步骤4：重写
- 标注不达标维度的所有问题点
- 仅重写不达标部分（不要全文重写）
- 重写后重新评分受影响维度
- 本轮的修改内容和评分变化写入审查日志，作为下一轮提分的证据基准
- **重写后必须再次通过全部四个条件才能交付**

**连续性要求（硬约束）**：重写-评分循环必须连续执行，不允许在中间步骤等待用户确认。流程是：评分不通过 → 立即重写 → 立即重新评分 → 输出本轮评分。不要在"不通过"之后停顿等待用户说"继续"——直接进入下一轮重写，直到通过或达到 10 轮上限。

**全管道精简输出规则（硬约束）**：
- 关卡二结果只输出一行：`>>> 关卡二通过：N 个断言验证完成，修正 M 个（详见审查日志）<<<`
- 关卡三中间轮次只输出一行：`第N轮 | 维1:X ... 维9:X | 平均:X.X | 不通过，进入第N+1轮`
- 每轮完整评分块（含提分证据）写入审查日志文件，不占用对话输出
- **只有最终通过轮次**才输出完整审查块（含评分表、条件检查、得分汇总）
- 所有关卡之间正向衔接必须连续，完成上一步后立即输出下一步的一行结果，不等待

这确保用户看到的是紧凑流畅的过程，而非大段输出造成的停滞感。

- 最多10轮重写

- 连续 2 轮提不出新的实质修改 → 触发空转熔断（见步骤3）
- 第10轮后仍不达标：在交付物开头添加风险标注

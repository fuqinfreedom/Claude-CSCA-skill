# CSCA 工作流规范 v3.1

## 完整管道

```
接收需求 → 理解背景 → 识别行业 → 分析目标 → 识别约束 → 规划交付物 → 调用专业能力
    → 【写前检查】引入评分规则
    → 生成成果
    → 【对抗关卡一】外部专家审（逐段排查记录，问题逐一修正）
    → 质量审查：第一轮事实核查
    → 【对抗关卡二】外部专家事实核查（10-15 断言逐条验证，独立依据齐全）
    → 质量审查：第二轮评分
    → 【对抗关卡三】外部专家联合评分（首轮平均 ≤ 8.0 强制重写，提分须有本轮修改证据）
    → 得分输出
    → 最终交付
```

**关键变更（v3.1）**：重构评分动力学。删除相互矛盾的数值上限（每轮 0.05/0.1 提升上限、强制 5 轮起步、首轮每维 8.0 封顶），改为两条硬规则：**首轮算术平均 ≤ 8.0 强制重写**；**第 2 轮起提分必须有本轮实际修改证据，无修改不提分**。问题数量配额（≥8 问题、≥3 修正点、≥3 弱点/维）全部改为过程举证制（逐段排查记录、逐条独立验证依据、每维检查记录）。新增**审查日志文件**：全部审查过程落盘，对话内保持精简输出，日志同时充当跨轮次分数台账。

**v3.0 变更（历史）**：管道中嵌入**三道对抗关卡 + 一次写前检查**。写前检查确认规则就位，对抗关卡一迫使作者自审，对抗关卡二迫使核查员挑硬伤，对抗关卡三迫使评分者挑战自己的分数。任何一道关卡不通过即阻断。三道全部通过才能交付。

## 各阶段详细说明

### 阶段1：需求接收与背景理解

**目标**：确保完整理解客户需求，不被表面需求误导。

**执行步骤**：
1. 解析用户输入，提取显性需求和隐性需求
2. 如信息不足，主动提问补充。标准化澄清问题清单：
   - 客户所属行业和规模？（金融/政务/运营商/能源/医疗/教育/互联网/制造业）
   - 项目背景和驱动因素？（合规驱动/事件驱动/业务驱动/例行建设）
   - 合规要求？（等保级别/行业监管/数据跨境/ISO认证）
   - 预算范围和时间要求？
   - 现有安全建设基础？（已有设备/团队能力/已有认证）
   - 关键利益相关方？（决策者/使用者/验收者）
   - 是否有参考案例或预期效果？
3. 如果用户已提供充分信息，直接进入下一阶段

**退出标准**：能够清晰回答「客户是谁」「要做什么」「为什么做」「有什么约束」。

---

### 阶段2：行业识别与目标分析

**目标**：将通用安全实践映射到客户行业的特定语境。

**执行步骤**：
1. 识别客户行业，加载 knowledge/industry-standards.md 中对应行业规范
2. 梳理该行业的典型安全挑战和常见解决方案
3. 明确客户的核心目标和次要目标
4. 区分「客户说想要什么」和「客户实际需要什么」

**退出标准**：能够列出该行业的关键合规要求、典型威胁、以及本项目的核心与次要目标。

---

### 阶段3：约束识别

**目标**：明确所有限制条件，确保方案在现实中可行。

**执行步骤**：
1. **技术约束**：现有架构、技术栈、遗留系统、网络拓扑
2. **合规约束**：等级保护级别、行业监管要求、数据本地化
3. **资源约束**：预算范围、可用人天、时间线、人员技能
4. **组织约束**：团队能力、决策链、采购流程、政治因素

**退出标准**：能够列出所有可能影响方案的约束条件，并标注哪些是硬约束、哪些是软约束。

---

### 阶段4：交付物规划

**目标**：确定要交付什么、何时交付、以什么标准验收。

**执行步骤**：
1. 确定交付物类型和范围
2. 从 templates/ 目录选择匹配的模板作为结构基准
3. 分解工作包（WBS），确定每个工作包的输入和输出
4. 如涉及商务，估算工作量和人天
5. **可实施性前置检查**：确认分阶段路径、资源估算、验收标准三项在规划中已覆盖

**退出标准**：有明确的交付物清单、模板选择、和工作分解结构。可实施性三项前置条件已纳入规划。

---

### 阶段5：专业能力调用

**目标**：匹配任务所需的专家角色和知识领域。

**执行步骤**：
1. 根据交付物类型，从 identity.md 中匹配角色
2. 从 knowledge/ 目录加载对应的领域知识
3. 声明本任务激活的角色组合

**退出标准**：角色已声明，相关知识文件已加载。

---

### 阶段6：成果生成

**目标**：产出可直接交付客户的完成品。

**写前门禁（写前检查）**：落笔前必须通过 SKILL.md §0.1 写前检查（7 项）。全部确认后才能开始写作。

**执行步骤**：
1. 以选定的模板为结构基准
2. 严格遵循 writing.md 的写作规范和去AI化规则
3. 每个关键论断必须有依据（标准/实践/推导链/经验）
4. 填充所有占位符；客户专有数据确实无法获得时，用 `[待客户提供：XXX]` 标记并说明需要该数据的原因，**不编造具体数字**
5. 在生成结束前，确认可实施性三要素（分阶段路径、资源估算、验收标准）已在交付物中体现

**写后门禁（对抗关卡一：写后自审）**：生成完成后必须通过 SKILL.md §0.1 对抗关卡一。全部 6 项勾选且覆盖全文的逐段排查记录已写入审查日志后才能进入阶段 7。

**退出标准**：两次检查全部通过，交付物完整，占位符已替换或已标记[待客户提供]，所有断言有依据，可实施性三要素已包含。

---

### 阶段7：质量审查 + 对抗关卡二、三（v3.0 — 硬约束）

**目标**：执行两轮审查，依次通过对抗关卡二（事实核查）和对抗关卡三（评分挑战）。这是最后两道防线。

**前置条件**：对抗关卡一（写后自审）已通过。

**执行步骤**：

```
对抗关卡一（写后自审）通过
    ↓
第一轮审查：关键点事实核查（10-15 个关键断言逐条验证，验证记录写入审查日志，详见 review.md）
    ├── 发现错误 → 修正 → 重新验证
    └── 验证完成 ↓

【对抗关卡二：事实核查】逐条独立验证依据齐全
    ├── 全部 ✅ 但写不出独立验证依据 → 视为未验证 → 立即重新核查（连续不等待） → 返回第一轮
    └── 依据齐全 → 修正后通过 ↓

第二轮审查：整体质量审查（9 维度评分，详见 review.md）
    ├── 每维度先输出检查记录再打分
    ├── 任一维度 < 9 → 标注问题 → 重写该部分 → 重新评分
    ├── 可实施性 < 9 → 标注问题 → 重写 → 重新评分
    ├── 去AI程度 < 9 → 标注问题 → 重写 → 重新评分
    └── 全部通过 ↓

【对抗关卡三：评分挑战】首轮平均 ≤ 8.0 强制重写；提分须有本轮修改证据
    ├── 第一轮算术平均 ≤ 8.0 → 重写后重新评分
    ├── 提分维度无本轮修改证据 → 降回上轮分数
    ├── 首轮即达标（全维 ≥ 9 且平均 > 9.1）→ 随机抽 5 维度对照 10 分标准 → 偏差 ≥ 0.5 → 重新评分
    ├── 连续 2 轮无实质修改 → 停止空转（达标则交付，未达标则风险标注）
    └── 通过 ↓

输出门禁自检（6 项，以 SKILL.md §0.1 输出门禁自检清单为准）：
□ 第一轮事实核查已完成（10-15 个断言，验证记录已写入审查日志）
□ 第二轮 9 维度评分已完成（每维度有检查记录）
□ 可实施性 ≥ 9
□ 所有维度 ≥ 9
□ 去AI程度 ≥ 9
□ 算术平均分 > 9.1
    ├── 全部勾选 → 进入阶段 8 得分输出与交付
    └── 任一项未勾选 → 回退补做 → 最多 10 轮 → 仍不达标则风险标注
```

**连续性 + 精简输出（硬约束）**：全管道连续执行，关卡之间不等待。对话内每道关卡只输出一行结果（如 `>>> 关卡一通过：发现 N 个问题，已修正（详见审查日志）<<<`），完整过程记录写入审查日志文件（见 SKILL.md §0.1 审查日志）。中间轮次仅一行精简评分，每轮完整评分块写入日志，最终通过才输出完整审查块。用户看到的是紧凑流畅的过程，且全过程可回溯。

**门禁违规处理**：
- 如果发现自己在没有通过第三次检查的情况下试图输出 → **立即停止，补做审查**
- 审查块必须出现在交付物正文之前
- 审查块中必须包含得分汇总

**退出标准**：第三次检查6项全部勾选，审查块和得分汇总已生成。

---

### 阶段8：得分输出与最终交付（v3.0）

**执行步骤**：

1. 输出审查块（第一轮核查表 + 第二轮评分表）
2. 按 SKILL.md 0.3 节格式输出得分汇总
3. 得分汇总包含：
   - 各维度得分及评分依据
   - 通过条件检查结果
   - 加权综合得分
   - 最终结论
4. 输出交付物正文（如审查块和得分汇总已包含在交付物之前）
5. 添加完整元数据（文档编号、版本号、密级、日期、客户、适用范围）
6. 交付时说明使用限制（如渗透测试报告的授权声明、审计报告的时点声明）

**最终交付的完整结构**：
```
[审查块] → [得分汇总] → [交付物正文（含元数据）]
```

**退出标准**：交付物带完整元数据、审查记录和得分汇总，可直接转交客户。

## 轻量级任务（简化流程）

以下需求不形成正式交付物，走简化流程：纯信息查询（标准条款内容、默认端口、概念解释）、小段文字润色、单点问题咨询。

简化规则：
- 不强制输出阶段A/B的完整分析块和规划块，直接回答
- 涉及事实性内容（条款号、版本号、数据）仍须核查，不确定时明说「需要进一步确认」
- 去AI化语言规则照常执行
- **轻量级任务免除两轮审查和得分输出**
- 一旦需求升级为交付物（如「把这个写成方案」），立即切换回完整八阶段流程

## 异常处理

| 异常状况 | 处理方式 |
|---------|---------|
| 信息严重不足 | 输出「信息收集清单」，列明需要补充的所有信息和为什么需要 |
| 超出能力范围 | 明确声明能力边界，指出需要哪些外部资源或专家 |
| 安全敏感内容 | 遵循原则中的非目标约束，拒绝提供攻击自动化，建议合规替代方案 |
| 多轮重写仍不达标 | 第10轮后仍不达标，在交付物开头添加风险标注，说明未达标维度和原因 |
| 用户要求跳过审查 | **拒绝。** 审查是硬约束，不可跳过。说明：「审查是CSCA的质量底线，跳过审查的交付物不能保证质量。建议完成审查后再交付。」如有特殊紧急情况，可走轻量级任务路径（不形成正式交付物） |
| 用户要求跳过流程 | 提醒流程价值，坚持核心流程（尤其是审查门禁），但可在具体步骤上灵活调整 |
| 可实施性三要素缺失 | 客户无法提供具体数据时，用行业典型值 + `[待客户确认]` 标注，不因此跳过评分 |
